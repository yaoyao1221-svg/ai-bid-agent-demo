# 企业资料抽取与本体入库 Skill

## 目标

把企业投标文件、营业执照、完税证明、资产负债表、产品资料、历史案例等资料，切片提取成标书精灵可复用的企业本体数据。

该 Skill 的首要原则是准确，不追求补得满。无法从文件确认的信息必须标记为“待 OCR/人工补充”，不能由 AI 猜测。

## 适用场景

- 用户提供企业投标文件，希望提取企业基本信息、商务信息、技术信息和案例信息。
- 用户提供营业执照、完税证明、资产负债表、产品资料，希望补全企业本体。
- 用户要求替换或更新 demo 中的企业样例数据。

## 输入文件类型

- `.docx`：优先用正文和表格抽取。
- `.pdf`：优先抽文本；扫描版需要 OCR。
- 图片扫描件：需要 OCR 后再入库。
- `.xlsx/.csv`：用于财务表、设备清单、案例清单。

## 标准流程

1. **文件切片**
   - 按标题、段落、表格拆分。
   - 每个切片保留来源文件、切片编号、页码或段落位置。
   - 大文件按 2500-4000 字符切片，避免一次性塞给模型。

2. **分类抽取**
   - 企业基本信息：名称、地址、联系人、电话。
   - 营业执照：登记机关、统一社会信用代码、注册资本、成立日期、经营范围、有效状态。
   - 资质证明：资质名称、等级、有效期、发证机关。
   - 完税证明：税款所属期、税种、证明编号、开具机关。
   - 财务信息：资产总额、负债总额、所有者权益、营业收入、利润等。
   - 设备信息：设备名称、型号、系统、关键参数、认证、质保。
   - 历史案例：项目名称、客户、年份、金额、行业、标签、交付内容。
   - 服务能力：本地服务、响应时效、质保、售后联系人。

3. **证据绑定**
   - 每条事实必须带 `evidence`。
   - 能定位页码/切片时，写清来源。
   - 只有目录出现、正文未出现细节的资料，标记为“材料存在，关键字段待补充”。

4. **保守校验**
   - 金额、日期、证书编号不能凭空补。
   - 扫描件未 OCR 时，只能确认“文件中疑似存在该材料”，不能确认具体字段。
   - 冲突信息进入 `extractionWarnings`，等待人工确认。

5. **本体入库**
   - 输出企业 JSON。
   - 同步生成抽取报告 Markdown。
   - 导入 demo 前先运行校验脚本。

## 输出结构

```json
{
  "name": "企业名称",
  "slogan": "企业能力摘要",
  "businessLicense": {
    "unifiedSocialCreditCode": "",
    "registeredAuthority": "",
    "registeredCapital": "",
    "establishedDate": "",
    "businessScope": "",
    "address": "",
    "status": "",
    "evidence": ""
  },
  "qualifications": [],
  "devices": [],
  "cases": [],
  "capabilities": [],
  "taxProofs": [],
  "financialStatements": [],
  "service": {},
  "contact": {},
  "pendingEvidence": [],
  "extractionWarnings": []
}
```

## 入库标准

- `qualifications` 至少包含一条可验证资质。
- `cases` 的金额和年份必须来自文件证据。
- `devices` 的核心参数必须来自产品响应、设备清单或认证材料。
- `pendingEvidence` 必须列出所有影响适配度和材料校验的缺口。
- 不允许把 `pendingEvidence` 中的信息当作已满足事实参与高置信匹配。

## 当前科讯资料缺口

- 营业执照统一社会信用代码、注册资本、成立日期、经营范围。
- 社保证明所属期间、参保人数。
- 完税证明所属期间、税种、证明编号。
- 资产负债表资产总额、负债总额、所有者权益。
- 产品认证证书编号和有效期。
- 更多历史项目案例。
- 售后响应时效的正式证明。
